# UI Slice 1: FastAPI-Based Patient Authoring Page

## Summary
Use a minimal FastAPI ASGI app for the new browser slice because the current repo environment already includes `fastapi`, `starlette`, `uvicorn`, and `python-multipart`, while the repo has no existing web app or template pattern and no `jinja2`. Render one server-side HTML page from the app module, keep JavaScript at zero, and route all submit handling through one tiny coordinator function that reuses the existing patient builder and mapper.

Repo-grounded decision:
- Existing repo web pattern: none found in `src/`, `tests/`, or docs.
- Existing installed lightweight web stack: `fastapi`, `starlette`, `uvicorn`.
- Existing template support: none found; `jinja2` is not installed.
- Therefore: use FastAPI, with inline server-rendered HTML for Slice 1.

## Key Changes
- Add a minimal FastAPI app with:
  - `GET /` redirecting to `/patient-authoring`
  - `GET /patient-authoring` rendering the empty form page
  - `POST /patient-authoring` validating input, running the authoring flow, and rerendering the same page with results or errors
- Keep orchestration out of the route body with one tiny function in the same module:
  - `run_patient_authoring_flow(narrative: str, complexity: str) -> PatientAuthoringPageResult`
  - It will:
    - trim and validate inputs
    - construct `PatientAuthoringInput`
    - call `build_patient_authored_record(...)`
    - call `map_authored_patient_to_patient_context(...)`
    - return a small typed result object for rendering
- Render the page with clearly separated labeled sections:
  - Input
  - Validation Errors
  - Authored Patient Profile
  - Mapped Patient Context
  - Raw JSON Inspection
- Render authored sections defensively so empty lists or absent facts show cleanly without breaking layout:
  - patient identity
  - conditions
  - medications
  - allergies
  - background facts
  - unresolved authoring gaps
  - authoring evidence
  - unmapped fields
- Use inline HTML in the FastAPI module because no repo template pattern or `jinja2` support exists today.
- Add direct `fastapi` and `uvicorn` runtime dependencies in `pyproject.toml` so this slice does not rely on transitive installs.

## Backend Route/Page Flow
- `GET /`
  - `302` or `307` redirect to `/patient-authoring`
- `GET /patient-authoring`
  - Return HTML page with heading, textarea, complexity select, submit button, and empty results containers
- `POST /patient-authoring`
  - Read form data
  - Validate:
    - trimmed narrative is not empty
    - complexity is one of `low`, `medium`, `high`
  - On invalid input:
    - return `400`
    - rerender same page
    - preserve entered values
    - show visible error section
  - On valid input:
    - call `run_patient_authoring_flow(...)`
    - rerender same page with authored record and mapped patient context sections populated
    - include raw JSON inspection blocks for both outputs

## Builder And Mapper Usage
The flow function will call the existing bounded authoring foundation directly:

```python
authoring_input = PatientAuthoringInput(
    authoring_text=narrative.strip(),
    complexity_level=complexity,
    scenario_label="patient-authoring-web-ui",
)
authored_record = build_patient_authored_record(authoring_input)
mapped_result = map_authored_patient_to_patient_context(authored_record)
```
