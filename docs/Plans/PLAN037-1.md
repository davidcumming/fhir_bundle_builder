7. Risks / notes

- The main risk is turning the wrapper into a second source of truth. It must stay a thin host over existing contracts.
- A second risk is duplicating orchestration logic inside executors. The wrapper should call existing authoring/orchestration helpers directly.
- A third risk is confusing the two workflows in Dev UI. Names and docs should clearly distinguish:
  - core deterministic bundle-builder workflow
  - authored-input demo workflow
- A fourth risk is hiding thin-provider limitations in the UI flow. The wrapper should surface gaps and unmapped facts honestly.
