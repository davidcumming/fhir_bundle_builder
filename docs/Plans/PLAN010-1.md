## 1. Repo assessment

- The current end-to-end workflow is structurally complete and already produces a deterministic content-enriched bundle for:
  - `Composition`
  - `Patient`
  - `MedicationRequest`
  - `AllergyIntolerance`
  - `Condition`
- Support resources are still thin in [resource_construction_builder.py](/Users/davidcumming/coding_projects/fhir_bundle_builder/src/fhir_bundle_builder/workflows/psca_bundle_builder_workflow/resource_construction_builder.py):
  - `Practitioner` remains base metadata only with deferred `name`
  - `Organization` remains base metadata only with deferred `name`
  - `PractitionerRole` currently only carries deterministic local references to `Practitioner` and `Organization`, with deferred `code`
- The current upstream inputs are narrower than the slice title suggests:
  - `NormalizedBuildRequest.provider_profile` only contains `profile_id`, `display_name`, and `source_type`
  - the schematic encodes the support-resource pattern and roles:
    - `composition-author -> practitionerrole-1`
    - `practitionerrole-practitioner -> practitioner-1`
    - `practitionerrole-organization -> organization-1`
    - placeholder roles: `author-practitioner`, `author-organization`, `document-author`
  - the PS-CA normalized asset layer contributes profile URLs and resource types, but no provider-organization directory data
- That means the repo can support deterministic enrichment for `Practitioner` now, but not meaningful organization identity or true provider-role context without expanding the provider input model.
- Current validation and repair are already positioned to absorb narrow support-resource checks:
  - workflow validation is deterministic and code-driven
  - repair routing already maps content failures to `resource_construction`
  - repair execution still does not rerun `resource_construction`, which is acceptable for this slice
- The main constraint now is to improve author/provider realism without inventing fake organization semantics from `provider_profile.display_name`.

## 2. Proposed slice scope

- Implement real deterministic enrichment for `Practitioner`.
- Implement only minimal, honest enrichment for `PractitionerRole`.
- Keep `Organization` intentionally thin in this slice.
- Recommended exact scope:
  - `Practitioner`
    - enrich from `NormalizedBuildRequest.provider_profile`
  - `PractitionerRole`
    - optionally add one deterministic author-role label derived from existing structured workflow artifacts, but do not pretend to know licensing, specialty, or organization affiliation details
  - `Organization`
    - do not invent name or identifier heuristics from the provider display label
- Reason for this boundary:
  - the repo has one provider label, not a provider plus organization plus role relationship model
  - using that single label for both `Practitioner` and `Organization`, or deriving an organization name from it, would bake in false semantics
  - `Practitioner` is the only support resource that can be meaningfully enriched from current inputs without architectural dishonesty
- Keep intentionally deferred after this slice:
  - real `Organization.name` / `Organization.identifier`
  - real `PractitionerRole` organizational context beyond references
  - provider-role relationship selection logic
  - bundle identifier/timestamp/fullUrl policy
  - broad provider master-data management

## 3. Proposed author/provider content-enrichment approach

- Keep all changes inside `resource_construction`; do not add a new stage.
- Do not expand `NormalizedBuildRequest` in this slice. The plan should explicitly acknowledge that meaningful support-resource enrichment beyond `Practitioner` depends on a future provider input model that includes organizations and provider roles.
- Recommended deterministic field policy now:
  - `Practitioner`
    - `active = true`
    - `identifier[0].value = provider_profile.profile_id`
    - `name[0].text = provider_profile.display_name`
    - keep richer name parts, telecom, qualification, and address deferred
  - `PractitionerRole`
    - keep existing `practitioner.reference` and `organization.reference`
    - add at most one narrow deterministic role-facing field:
      - `code[0].text = "document-author"`
      - derive from `ResourcePlaceholder.role` / current schematic author pattern, not from free text
    - keep specialty, telecom, period, and availability deferred
  - `Organization`
    - no fake name or identifier population
    - remain base metadata only for now
- Provenance should keep using the existing `DeterministicValueEvidence` model:
  - `Practitioner.identifier[0].value` from `normalized_request.provider_profile.profile_id`
  - `Practitioner.name[0].text` from `normalized_request.provider_profile.display_name`
  - `Practitioner.active` from a fixed deterministic content policy
  - `PractitionerRole.code[0].text` from `bundle_schematic.resource_placeholders[practitionerrole-1].role` if that field is added
- Validation should stay narrow and only protect what is truly added:
  - add `bundle.practitioner_identity_content_present`
  - add `bundle.practitionerrole_author_context_present` only if `PractitionerRole.code[0].text` is introduced
  - do not add a new `Organization` content rule in this slice
- Repair routing should map any new support-resource content checks to `resource_construction`.
- Keep naming honest in stage summaries, assumptions, and docs:
  - `Practitioner` is enriched from the currently selected provider stub
  - `Organization` and richer `PractitionerRole` semantics remain deferred pending expanded provider inputs

## 7. Risks / notes

- The main risk is pretending the current single `provider_profile.display_name` can represent both a person and an organization. This slice should not do that.
- A second risk is enriching `PractitionerRole` too aggressively. Without explicit provider-role input data, it should stay limited to references and, at most, one narrow author-role placeholder label.
- The workflow may still look support-resource-light because `Organization` remains thin. That is acceptable if the artifacts make the reason explicit.
- Because repair execution still does not retry `resource_construction`, any new support-resource repair routes remain recommendation-only for now. That is consistent with the current architecture.
