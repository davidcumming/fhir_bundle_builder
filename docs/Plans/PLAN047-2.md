## Key Changes
- Add a minimal FastAPI app with:
  - `GET /` redirecting to `/patient-authoring`
  - `GET /patient-authoring` rendering the empty form page
  - `POST /patient-authoring` handling form parsing, validation, flow execution, and rerendering
- Keep route validation explicit in the `POST` handler:
  - read form data
  - trim narrative
  - validate non-empty narrative
  - validate complexity in `low|medium|high`
  - on failure, return `400` with sticky values and visible errors
- Keep orchestration out of the route body with one tiny coordinator function in the same module:
  - `run_patient_authoring_flow(narrative: str, complexity: str) -> PatientAuthoringPageResult`
  - responsibilities only:
    - construct `PatientAuthoringInput`
    - call `build_patient_authored_record(...)`
    - call `map_authored_patient_to_patient_context(...)`
    - return a small typed result for rendering
- Use a minimal typed result object:
  - tiny `dataclass` or equivalently lightweight structure
  - no service class, no abstraction layer, no extra module unless clearly needed
- Render one page with clearly separated labeled sections for visual inspection:
  - Submitted Input
  - Validation Errors
  - Authored Patient Profile
  - Mapped Patient Context
  - Raw JSON Inspection
- Keep HTML inline in the FastAPI module unless implementation uncovers an existing maintainable template pattern that was missed.

## Backend Route/Page Flow
- `GET /`
  - Redirect to `/patient-authoring`
- `GET /patient-authoring`
  - Render HTML page with:
    - heading `Patient Profile Authoring`
    - textarea for patient narrative
    - select for `low`, `medium`, `high`
    - submit button
    - empty but clearly labeled result sections or placeholders
- `POST /patient-authoring`
  - Read form data
  - Trim the narrative
  - Validate:
    - narrative is not empty after trim
    - complexity is one of `low`, `medium`, `high`
  - On validation error:
    - return `400`
    - preserve entered narrative and selected complexity
    - render visible Validation Errors section
    - do not call the coordinator function
  - On success:
    - call `run_patient_authoring_flow(...)`
    - rerender the same page with populated Submitted Input, Authored Patient Profile, Mapped Patient Context, and Raw JSON Inspection sections

## Builder And Mapper Usage
The coordinator function will directly reuse the existing authoring foundation:

```python
authoring_input = PatientAuthoringInput(
    authoring_text=narrative,
    complexity_level=complexity,
    scenario_label="patient-authoring-web-ui",
)
authored_record = build_patient_authored_record(authoring_input)
mapped_result = map_authored_patient_to_patient_context(authored_record)
```

The route is responsible for trimming and validating before this function is called.

## Rendering Requirements
The page must clearly separate and label these sections:
- Submitted Input
- Validation Errors
- Authored Patient Profile
- Mapped Patient Context
- Raw JSON Inspection

The authored result area should render, when present:
- patient identity
- conditions
- medications
- allergies
- background facts
- unresolved authoring gaps
- authoring evidence
- unmapped fields

Rendering should be resilient when sections are empty:
- empty lists should render as empty-state text, not disappear silently
- missing optional fields should render cleanly without breaking layout
- validation errors must be visually distinct from successful result content
